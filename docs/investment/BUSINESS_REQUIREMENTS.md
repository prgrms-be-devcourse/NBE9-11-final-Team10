## 1.1 종목 다건 조회 - 텍스트 검색 기반

- 거래 가격

# 3. 주식 거래

## User Story

사용자는 종목 상세 조회 화면에서 실시간 시세를 보며 주문을 수행할 수 있다.

---

## 공통 정책

시장가 주문과 지정가 주문 모두 실제 체결은 다음 조건을 모두 만족해야 한다.

- 정규장 시간
- 개장일
- 거래정지 종목 아님

---

### 주문 가능 여부 검증

주문 요청 시 다음을 검증한다.

- 계좌 비밀번호 일치 여부
- 예수금 충분 여부
- 보유 수량 충분 여부 (매도)
- 중복 요청 여부

---

### 중복 요청 방지 정책

주문 요청은 `idempotencyKey`를 필수로 전달한다.

`idempotencyKey`는 UUID 형식을 사용한다.

동일한 논리적 주문 요청에 같은 `idempotencyKey`를 재사용하는 책임은 클라이언트에 있다.

예시

```text
주문 폼 생성 또는 주문 의도 생성
↓
UUID 생성
↓
주문 요청
↓
응답 타임아웃 또는 네트워크 재시도
↓
동일 UUID로 재요청
```

백엔드는 다음 방식으로 멱등성을 보장한다.

- `idempotencyKey` 필수 검증
- UUID 형식 검증
- `investment_account_id + idempotency_key` 유니크 제약
- 동일 키 요청에 대해 신규 주문 생성 금지

프론트엔드는 더블 클릭 방지를 위해 주문 요청 중 버튼 비활성화를 적용한다.

단, 클라이언트가 동일한 주문 의도에 대해 서로 다른 UUID를 발급하면 백엔드는 동일 요청 여부를 확정할 수 없다.

---

### 공통 고려사항

- 계좌 동시성 제어
- 주문 동시성 제어
- 체결 동시성 제어
- DB 데드락 방지
- 트랜잭션 최소화
- 비동기 처리 가능성 검토

---

### 주문 실패 사유

다음 상황에서는 주문을 거부한다.

- 휴장일
- 정규장 아님
- 거래정지 종목
- 계좌 비밀번호 불일치
- 예수금 부족
- 보유 수량 부족
- 중복 요청

---

## 3.1 시장가 즉시 주문

### User Story

사용자는 현재 시장 가격 기준으로 즉시 주문을 수행할 수 있다.

---

### 기능 요구사항

- 요청 즉시 체결을 시도한다.
- 실시간 호가 정보를 기반으로 체결한다.
- 최선 호가부터 순차적으로 수량을 소진한다.

---

### 체결 방식

주문 요청 수량에 대해 실시간 호가 수량을 기준으로 가능한 최대 수량까지 체결한다.

---

### 잔량 부족 시

모든 호가 수량보다 주문 요청 수량이 더 많을 경우

- 가능한 수량만 체결
- 나머지 수량은 취소

처리한다.

---

### 체결 완료 후 처리

- 계좌 잔고 갱신
- 보유 종목 갱신
- 주문 내역 저장
- 체결 내역 저장

---

## 3.2 지정가 주문 예약

### User Story

사용자는 원하는 가격으로 예약 주문을 등록할 수 있다.

---

### 기능 요구사항

- 지정가 주문 등록은 24시간 가능하다.
- 실제 체결은 정규장 시간에만 가능하다.
- 지정가 주문 정보를 별도 저장한다.

---

### 주문 상태

- PENDING
- PARTIALLY_FILLED
- FILLED
- CANCELLED

---

### 체결 방식

실시간 KIS WebSocket으로 수신한 호가 이벤트를 기반으로 체결을 수행한다.

호가가 발생할 경우

- 주문 조건 확인
- 체결 가능 여부 검증
- 수량 검증
- 주문 체결

을 수행한다.

---

### 부분 체결

예시

100주 주문

40주 체결

60주 잔여

→ PARTIALLY_FILLED

---

체결 가능한 수량만 우선 체결한다.

잔여 수량은 지정가 주문 테이블에 유지한다.

---

### 체결 우선순위

동일 가격 조건의 지정가 주문이 여러 건 존재할 경우

주문 생성 시각이 빠른 주문부터 체결한다.

(FIFO)

---

### 지원 기능

- 주문 등록
- 주문 수정
- 주문 취소

---

### 주문 수정 가능 조건

- PENDING 상태
- PARTIALLY_FILLED 상태

---

### 주문 취소 가능 조건

- PENDING 상태
- PARTIALLY_FILLED 상태

---

### 수정 및 취소 불가 상태

- FILLED
- CANCELLED

---

### 향후 개선 사항

스케줄링 기반 Polling 방식보다 이벤트 기반 처리 방식을 우선 고려한다.

후보 기술

- Redis Pub/Sub
- RabbitMQ
- Kafka

---

필요 시 조건을 만족하는 예약 주문들에 대해

- 병렬 처리
- 비동기 처리

를 적용하여 성능 향상을 검토한다.
