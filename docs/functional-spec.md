# 기능 명세서 - Banking MVP

## 1. 문서 목적

Team10 Banking MVP에서 구현할 최소 핵심 기능의 범위, 사용자 흐름, 데이터, 정책, 예외 상황을 정의한다. 이 문서는 백엔드 구현, 프론트엔드 연동, 테스트 케이스 작성의 기준으로 사용한다.

## 2. MVP 범위

| 도메인 | 기능 | MVP 포함 | 설명 |
| --- | --- | --- | --- |
| Auth/User | 회원가입 | 포함 | 사용자 기본 정보와 로그인 정보를 등록한다. |
| Auth/User | 로그인 | 포함 | 이메일/비밀번호로 인증하고 토큰을 발급한다. |
| Auth/User | 사용자 정보 조회 | 포함 | 로그인한 사용자의 프로필/인증 상태를 조회한다. |
| Auth/User | 본인인증 | 포함 | 계좌 개설/송금 전 사용자 본인인증 상태를 완료 처리한다. |
| Account | 계좌 개설 | 포함 | 인증된 사용자가 신규 계좌를 개설한다. |
| Account | 내 계좌 목록/상세 조회 | 포함 | 사용자가 보유한 계좌와 잔액을 조회한다. |
| Transfer | 입금 | 포함 | 본인 계좌에 금액을 입금한다. |
| Transfer | 계좌 간 송금 | 포함 | 내 계좌에서 다른 계좌로 금액을 이체한다. |
| Transaction | 거래내역 조회 | 포함 | 계좌별 거래내역을 다건/단건 조회한다. |
| Transaction | 거래내역 필터링 | 포함 | 기간, 입출금 방향, 거래액, 거래 상대 기준으로 필터링한다. |
| Transaction | 과거 거래내역 기록 테이블 | 포함 | 입금/송금 발생 시 불변 거래 기록을 저장한다. |

## 3. 공통 정책

### 3.1 인증/인가

- 회원가입과 로그인은 비로그인 상태에서 접근 가능하다.
- 사용자 정보 조회, 본인인증, 계좌, 입금/송금, 거래내역 API는 로그인 토큰이 필요하다.
- 사용자는 본인 소유 계좌만 조회/입금/출금할 수 있다.
- 송금 수취 계좌는 타인 계좌일 수 있으나, 출금 계좌는 반드시 본인 소유여야 한다.

### 3.2 금액 정책

- 모든 금액은 원화 정수(`KRW`, `Long`)로 처리한다.
- 입금/송금 금액은 `1원 이상`이어야 한다.
- 송금 시 출금 계좌 잔액이 송금 금액보다 작으면 실패한다.
- 잔액 변경과 거래내역 저장은 하나의 트랜잭션으로 처리한다.

### 3.3 계좌 정책

- 계좌번호는 서버에서 생성하며 중복될 수 없다.
- MVP 계좌 상태는 `ACTIVE`, `CLOSED`, `SUSPENDED`를 사용한다.
- `ACTIVE` 계좌만 입금/송금이 가능하다.
- 거래내역 조회는 계좌 소유자 검증을 수행한다.
- 계좌 개설은 본인인증 완료 사용자만 가능하다.

### 3.4 거래내역 정책

- 입금/송금 성공 시 거래내역을 반드시 생성한다.
- 거래내역은 정정/삭제하지 않는 것을 원칙으로 한다.
- 송금은 출금 계좌와 입금 계좌에 각각 `type=TRANSFER`, `direction=OUT/IN` 거래내역을 생성한다.
- 단순 입금은 `type=DEPOSIT`, `direction=IN` 거래내역을 생성한다.
- 거래 전/후 잔액(`balanceBefore`, `balanceAfter`)을 기록해 과거 내역 조회 시 현재 잔액과 무관하게 당시 상태를 확인할 수 있게 한다.
- 이체 거래는 상대 계좌번호(`counterpartyAccountNumber`)와 상대 이름(`counterpartyName`)을 당시 값으로 저장한다.

## 4. 도메인별 기능 명세

## 4.1 Auth/User

### 4.1.1 회원가입

**목표**  
신규 사용자가 서비스 이용을 위한 계정을 생성한다.

**입력 정보**

- 이메일
- 비밀번호
- 이름
- 휴대폰 번호
- 생년월일

**처리 절차**

1. 필수값과 형식을 검증한다.
2. 이메일 중복 여부를 확인한다.
3. 비밀번호를 해시 처리해 저장한다.
4. 사용자 상태를 생성한다.
5. 본인인증 상태는 기본값 `false`로 저장한다.

**성공 결과**

- 사용자 ID가 생성된다.
- 사용자는 로그인할 수 있다.

**예외**

- 이메일 형식 오류
- 비밀번호 정책 불만족
- 이미 가입된 이메일
- 필수값 누락

### 4.1.2 로그인

**목표**  
등록된 사용자가 이메일/비밀번호로 인증하고 API 호출용 토큰을 발급받는다.

**입력 정보**

- 이메일
- 비밀번호

**처리 절차**

1. 이메일로 사용자를 조회한다.
2. 비밀번호를 검증한다.
3. 인증 성공 시 access token을 발급한다.

**성공 결과**

- access token 반환
- 사용자 기본 정보 반환

**예외**

- 존재하지 않는 사용자
- 비밀번호 불일치

### 4.1.3 사용자 정보 조회

**목표**  
로그인한 사용자의 기본 정보와 본인인증 상태를 조회한다.

**조회 정보**

- 사용자 ID
- 이메일
- 이름
- 휴대폰 번호
- 생년월일
- 본인인증 여부
- 생성일시

### 4.1.4 본인인증

**목표**  
계좌 개설과 송금 가능 조건인 본인인증을 완료한다.

**MVP 처리 방식**

- 외부 본인인증 연동 없이 요청값 검증 후 인증 완료 처리한다.
- 추후 PASS/KCB/NICE 등 외부 인증 연동으로 교체 가능하게 서비스 계층을 분리한다.

**입력 정보**

- 이름
- 휴대폰 번호
- 생년월일
- 인증 코드 또는 인증 토큰

**성공 결과**

- 사용자의 `identityVerified` 값이 `true`가 된다.

**예외**

- 사용자 정보 불일치
- 이미 인증 완료
- 인증 코드 만료/불일치

## 4.2 Account

### 4.2.1 계좌 개설

**목표**  
본인인증을 완료한 사용자가 입금/송금에 사용할 계좌를 개설한다.

**입력 정보**

- 계좌 별칭
- 상품명 또는 계좌 타입

**처리 절차**

1. 로그인 여부를 확인한다.
2. 사용자 본인인증 완료 여부를 확인한다.
3. 계좌번호를 생성한다.
4. 초기 잔액 `0원`, 상태 `ACTIVE`로 계좌를 저장한다.

**성공 결과**

- 신규 계좌 ID/계좌번호/잔액/상태 반환

**예외**

- 미인증 사용자
- 계좌번호 생성 실패 또는 중복

### 4.2.2 내 계좌 목록 조회

**목표**  
로그인 사용자가 본인 계좌 목록과 요약 정보를 확인한다.

**조회 정보**

- 계좌 ID
- 계좌번호
- 계좌 별칭
- 잔액
- 상태
- 생성일시

### 4.2.3 내 계좌 상세 조회

**목표**  
특정 계좌의 상세 정보와 현재 잔액을 확인한다.

**예외**

- 존재하지 않는 계좌
- 타인 소유 계좌 접근

## 4.3 Transfer

### 4.3.1 입금

**목표**  
사용자가 본인 계좌에 금액을 입금한다.

**입력 정보**

- 입금 대상 계좌 ID 또는 계좌번호
- 금액
- 메모

**처리 절차**

1. 대상 계좌가 본인 소유인지 확인한다.
2. 대상 계좌 상태가 `ACTIVE`인지 확인한다.
3. 금액이 1원 이상인지 검증한다.
4. 계좌 잔액을 증가시킨다.
5. `type=DEPOSIT`, `direction=IN` 거래내역을 저장한다.

**성공 결과**

- 입금 처리 결과
- 입금 후 잔액
- 거래내역 ID

**예외**

- 존재하지 않는 계좌
- 타인 계좌 직접 입금 시도
- 비활성 계좌
- 유효하지 않은 금액

### 4.3.2 계좌 간 송금

**목표**  
사용자가 본인 출금 계좌에서 수취 계좌로 금액을 송금한다.

**입력 정보**

- 출금 계좌 ID
- 수취 계좌번호
- 금액
- 메모

**처리 절차**

1. 출금 계좌가 본인 소유인지 확인한다.
2. 출금 계좌와 수취 계좌가 존재하고 `ACTIVE`인지 확인한다.
3. 금액이 1원 이상인지 검증한다.
4. 출금 계좌 잔액이 충분한지 확인한다.
5. 출금 계좌 잔액을 감소시킨다.
6. 수취 계좌 잔액을 증가시킨다.
7. 출금 계좌에 `type=TRANSFER`, `direction=OUT` 거래내역을 저장한다.
8. 수취 계좌에 `type=TRANSFER`, `direction=IN` 거래내역을 저장한다.
9. 송금 결과를 저장한다.

**성공 결과**

- 송금 ID
- 송금 상태 `SUCCESS`
- 출금 후 잔액
- 수취 계좌 정보 일부

**예외**

- 출금 계좌 없음
- 수취 계좌 없음
- 타인 출금 계좌 접근
- 잔액 부족
- 비활성 계좌
- 유효하지 않은 금액

## 4.4 Transaction

### 4.4.1 거래내역 다건 조회

**목표**  
사용자가 본인 계좌의 과거 거래내역을 필터링과 페이징 조건으로 조회한다.

**필터 조건**

- 계좌 ID: 필수
- 시작일/종료일: 선택
- 입출금 방향: `IN`, `OUT` 선택
- 최소/최대 금액: 선택
- 거래 상대 이름: 선택, 최대 30자
- 페이지 번호: 선택, 기본값 `0`, 0 이상
- 정렬 방향: 선택, 기본값 `DESC`, `ASC` 또는 `DESC`

**페이징/정렬 정책**

- 한 페이지 크기는 `20`건으로 고정한다.
- 정렬 컬럼은 `transactedAt` 하나만 허용한다.
- 사용자는 정렬 방향만 요청할 수 있다.

**조회 정보**

- 거래내역 ID
- 거래 상대 이름
- 금액
- 거래 후 잔액
- 거래일시
- 입출금 방향

**예외**

- 계좌 소유자 검증 실패
- 잘못된 기간 조건
- 잘못된 금액 범위 조건

### 4.4.2 거래내역 단건 조회

**목표**  
사용자가 본인 계좌에 속한 특정 거래내역의 상세 정보를 조회한다.

**입력 정보**

- 계좌 ID: 필수
- 거래내역 ID: 필수

**조회 정보**

- 거래내역 ID
- 거래 유형
- 입출금 방향
- 금액
- 거래 후 잔액
- 거래 상대 이름
- 메모
- 거래일시

**예외**

- 계좌 소유자 검증 실패
- 해당 계좌에 속하지 않는 거래내역
- 존재하지 않는 거래내역

## 6. MVP 완료 기준

- 사용자가 회원가입 후 로그인할 수 있다.
- 로그인 사용자가 본인 정보를 조회할 수 있다.
- 사용자가 본인인증을 완료할 수 있다.
- 본인인증 완료 사용자가 계좌를 개설할 수 있다.
- 사용자가 본인 계좌에 입금할 수 있다.
- 사용자가 본인 계좌에서 다른 계좌로 송금할 수 있다.
- 입금/송금 성공 시 거래내역이 기록된다.
- 사용자가 본인 계좌의 거래내역을 필터링해 다건 조회하고, 특정 거래내역을 단건 조회할 수 있다.
